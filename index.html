PIOPD turns posterior behavior into online preference

The generator deployed online cannot observe clicked target queries or future engagement. PIOPD uses those posterior signals only during training, distilling them into the model weights so the same low-latency student can serve with trigger-side evidence alone.

Serving constraint

Built for the 20-30 ms bottom-bar window

Bottom-bar recommendation must run after the feed item is chosen, yet the query needs to be visible as the user starts watching. OneBar keeps the same preparation stage and collapses the online cascade into a single generation stage, reducing total online processing while enabling context-aware query generation.

How PIOPD works

PIOPD is the paper's central training contribution. It projects a posterior-aware teacher onto the deployable student policy over the states the student actually visits, making preference learning both behavior-grounded and serving-compatible.

Core contribution

Preference internalization instead of reward-model reranking

Many generative recommendation systems align outputs through a separately trained reward model. PIOPD removes that extra online dependency by using posterior behavior evidence as privileged training-time supervision.

  • 1Teacher sees posterior evidence: clicked target queries expose the behavior-confirmed intent region.
  • 2Student keeps the serving input: it receives only trigger-side evidence available online.
  • 3Distillation happens on-policy: teacher and student are compared on student-generated prefixes.

Key ideas

The framework is designed around noisy short-video metadata, sparse behavior evidence, and production safety requirements.

PIOPD

Posterior preferences become model weights

A teacher with clicked-query evidence supervises the online student on its own generated states, letting behavior-confirmed preferences survive after the teacher is removed.

Grounding

Collaborative-multimodal intent evidence

OneBar combines cleaned textual fields, multimodal video summaries, behavior-derived query anchors, and trigger-relevant user history before generation.

Generation

Compact end-to-end query generator

A BART encoder-decoder consumes a short field-aligned schema and generates the bottom-bar query list directly, with lightweight post-checks for serving.

Method overview

OneBar serializes heterogeneous evidence into a compact prompt, generates with a single encoder-decoder model, and learns preferences progressively from content-grounded supervision to posterior on-policy distillation.

Progressive training around PIOPD

PIOPD is most effective after the generator has learned content-grounded query language and coarse behavior orderings. The three-stage path prepares the student distribution, then applies dense posterior distillation.

1

Content-grounded SFT

Clicked query targets and rejection supervision teach the generator when to recommend and when to abstain.

2

List-wise feedback alignment

Hierarchical behavior levels organize clicked, engaged, exposed, and retrieved candidates into ordinal preference lists.

3

PIOPD

A posterior-aware frozen teacher supplies token-level distributions on student rollouts, stabilized with bidirectional KL, entropy regularization, R-Drop, and FGM.

FAQ

Is PIOPD the same as OPSD?

PIOPD is an OPSD-style posterior preference distillation strategy for generative recommendation. It adapts the on-policy distillation loop to a teacher with posterior behavior evidence and a deployable trigger-only student.

Why not just use retrieval?

Retrieval-based pipelines are bounded by historical candidate pools and fragmented objectives. OneBar can compose fresh, fine-grained queries from current content evidence and collaborative anchors.

How does safety work?

The generator can emit a special rejection token when evidence is weak or risky. Online lightweight checks still remain, but the model itself learns abstention behavior.
